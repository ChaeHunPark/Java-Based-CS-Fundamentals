### 🔍 핵심 통찰: TCP 3-Way Handshake
| 단계 | 플래그 | 의미 | 비유 |
|:---:|:---:|:---|:---|
| 1 | **SYN** | 클라이언트 -> 서버 연결 요청 | "저기요, 대화 가능하신가요?" |
| 2 | **SYN-ACK** | 서버 -> 클라이언트 응답 및 요청 | "네, 들립니다. 제 말도 들리세요?" |
| 3 | **ACK** | 클라이언트 -> 서버 최종 응답 | "네, 잘 들립니다. 말씀하시죠!" |

> **주의**: 이 과정은 애플리케이션(Java)이 아닌 운영체제(OS)의 TCP 스택에서 자동으로 수행됩니다. `accept()`가 리턴되었다면 이 과정이 이미 성공적으로 끝났음을 의미합니다.

- [x] **Step 02: HTTP Request Header 분석** [코드](./Step02_HttpServer.java)
    - **내용**: TCP 소켓 위에 HTTP 프로토콜 규격을 얹어 브라우저(Chrome)와 직접 통신 구현.
    - **핵심 통찰**:
        1. **HTTP는 텍스트다**: 브라우저와 서버는 정해진 규격(Method, Header, Body)의 텍스트를 주고받음으로써 동작함.
        2. **다중 요청의 원리**: 브라우저는 메인 페이지(`/`)를 받은 후, 아이콘(`favicon.ico`) 등을 가져오기 위해 별도의 TCP 연결을 추가로 시도함.
        3. **Stateless (무상태성)**: 서버는 각 요청이 동일한 사용자인지 기본적으로 알지 못하며, 헤더 정보를 통해 상태를 추론함.
        4. **응답 규격의 중요성**: `HTTP/1.1 200 OK` 상태 라인과 헤더-본문 사이의 **'빈 줄(CRLF)'**이 있어야만 브라우저가 정상 렌더링 가능.

## 💡 네트워크 핵심 용어

| 용어 | 설명 | 비유 |
| :--- | :--- | :--- |
| **IP (Internet Protocol)** | 컴퓨터의 고유 주소 (3계층) | 택배를 보낼 때의 집 주소 |
| **Port** | 한 컴퓨터 내의 프로세스 식별 번호 (4계층) | 아파트 호수 (8080호, 3306호 등) |
| **TCP** | 연결 지향형, 신뢰성 보장 (분실 시 재전송) | 전화 통화 (상대방과 연결 확인 후 대화) |
| **UDP** | 비연결형, 속도 우선 (분실 감수) | 방송/스트리밍 (상대가 보든 말든 송출) |
| **Handshake** | 데이터를 주고받기 전 연결을 확립하는 과정 | 대화 시작 전 "저기요?", "네?" 하는 절차 |
| **Keep-Alive** | 한 번 맺은 TCP 연결을 끊지 않고 재사용하는 옵션 | 볼일 끝날 때마다 전화 끊지 않고 계속 들고 있기 |
| **User-Agent** | 클라이언트의 브라우저 및 OS 정보 헤더 | 서버에 제출하는 브라우저의 신분증 |
